# 评论分类 Agent 项目技术说明文档 (报告参考)

本方案旨在利用大语言模型 (LLM) 和 Agent 架构，实现对电商评论的细粒度情感分析 (ABSA - Aspect-Based Sentiment Analysis)。相比传统机器学习，本项目通过“检索增强 (RAG)”和“自我修正 (Reflector)”机制，显著提升了分类的准确率和召回率。

***

## 1. 技术栈概览 (Technical Stack)

| 维度         | 技术选型                        | 作用                                         |
| :--------- | :-------------------------- | :----------------------------------------- |
| **核心大脑**   | LLM                         | 负责语义理解、实体提取和逻辑推理。                          |
| **工作流框架**  | **LangGraph**               | 构建有状态的、支持循环迭代的 Agent 工作流。                  |
| **向量数据库**  | **ChromaDB**                | 存储原始评论，支持语义搜索 (RAG) 以获取相似案例。               |
| **关系型数据库** | **MySQL**                   | 存储标准标签数据 (Ground Truth)，提供高质量 Few-shot 示例。 |
| **前端交互**   | **Streamlit**               | 提供简单的网页端对话界面，支持思考过程实时展示。                   |
| **并发处理**   | Python `ThreadPoolExecutor` | 在批量测试阶段提供高效的并行处理能力。                        |

***

## 2. 数据库设计 (Database Design)

本项目采用 **双库架构**，分别应对非结构化语义检索和结构化标签存储的需求。

### 2.1 向量数据库 (ChromaDB)

- **存储内容**：原始评论文本及其对应的 Embedding。
- **作用**：当输入一条新评论时，计算其语义向量，从库中检索出最相似的 3-5 条历史评论。
- **目的**：通过 RAG (检索增强生成) 为模型提供“参考范例”，解决长尾场景分类难的问题。

### 2.2 关系型数据库 (MySQL)

- **表结构**：`labels` 表。
- **核心字段**：`id` (评论唯一标识), `AspectTerms` (属性词), `OpinionTerms` (观点词), `Categories` (类别), `Polarities` (极性)。
- **作用**：通过 ID 关联向量库检索出的结果，提取对应的标准答案，构造 Few-shot Prompt。

***

## 3. 模块化 Agent 设计 (Module Design)

项目基于 **LangGraph** 构建了一个闭环工作流，包含三个核心节点：

### 3.1 检索节点 (Retrieval Node)

- **输入**：用户评论。
- **操作**：
  1. 在 ChromaDB 中进行向量检索。
  2. 根据检索到的 ID，在 MySQL 中查询对应的标注结果。
- **输出**：包含相似评论及其标签的示例集 (Examples)。

### 3.2 分类节点 (Classification Node)

- **核心逻辑**：基于 Few-shot 提示词，要求模型提取属性词和观点词。
- **约束条件**：**词源一致性**。强制要求提取的词必须 100% 存在于原句中（严禁归一化或改写）。
- **重试机制**：如果收到评分节点的反馈，该节点会根据错误信息进行针对性修正。

### 3.3 评分/评估节点 (Evaluation Node)

- **硬约束检查**：程序化校验模型输出是否符合 JSON 格式、词源是否一致、类别是否在合法枚举中。
- **逻辑评估**：利用 LLM 检查是否漏掉关键信息。
- **反馈循环**：若校验失败，生成具体的错误描述 (Feedback) 并打回分类节点；若成功或达到重试上限，则进入终态。

***

## 4. 核心亮点 (Key Features)

1. **Reflector (反射/修正) 模式**：模型不是一次性输出结果，而是通过“分类-评估-报错-再分类”的循环，不断逼近正确答案。
2. **多并发安全**：通过单例模式 (Singleton) 封装 RAG 服务，配合线程锁 (threading.Lock)，解决了并发模式下文件被占用或写入冲突的问题。
3. **思考过程可视化**：在交互 Demo 中，用户可以看到 Agent 内部的思考步骤（如“正在检索相似案例”、“硬约束校验失败，正在重试”等），增强了系统的可解释性。
4. **稳定排序算法**：在最终提交结果生成阶段，使用 `kind='stable'` 排序算法，确保多标签评论在 ID 排序后，其内部的相对顺序不会乱序。

## 5. 项目文件结构参考

- `agent.py`: 定义 LangGraph 图结构和状态流转。
- `app.py`: Streamlit 交互界面实现。
- `test_run.py`: 批量并发测试脚本。
- `utils/node.py`: Agent 节点具体业务逻辑（最核心）。
- `utils/tools.py`: 数据库交互服务。
- `rag/ragservice.py`: 向量数据库封装。
